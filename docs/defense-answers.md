# Ответы на контрольные вопросы защиты

## Стили интеграции

В проекте используется синхронная интеграция через REST API и orchestration pattern в IntegrationService. Для production-расширения возможны event-driven integration, messaging через RabbitMQ/Kafka, shared database integration запрещается для микросервисов из-за сильной связности.

## BPMN

BPMN описывает бизнес-процесс: выбор ОС, получение карточки, расчет амортизации, принятие решения, списание и фиксация результата. BPMN отвечает на вопрос "как идет бизнес-процесс", а UML deployment/component отвечает на вопрос "какими техническими компонентами он реализован".

## Swagger/OpenAPI

Swagger/OpenAPI используется как машинно-читаемый контракт REST API. В ASP.NET Core подключены `AddSwaggerGen()`, `UseSwagger()` и `UseSwaggerUI()`. XML-комментарии попадают в OpenAPI-документацию и помогают frontend/backend-разработчикам синхронизировать контракт.

## Saga

Saga нужна для распределенных бизнес-транзакций, где нельзя использовать обычную ACID-транзакцию между микросервисами. В сценарии списания ОС Saga может хранить шаги: резервирование операции, расчет амортизации, списание, аудит. При сбое запускаются компенсации: отменить операцию, снять резерв, уведомить оператора.

## Polly

Polly реализует отказоустойчивость HTTP-вызовов: retry с exponential backoff для временных ошибок и circuit breaker для защиты от повторных отказов downstream-сервиса. Это снижает риск каскадного падения микросервисов.

## Serilog

Serilog используется для структурированного логирования. В проекте настроены console sink и rolling file sink. Логи содержат correlation id, inventory id и технический контекст, что помогает расследовать ошибки интеграционного потока.

## Redis

Redis используется как distributed cache. IntegrationService кэширует список объектов ОС, чтобы уменьшить количество обращений к ObjectService и ускорить UI. TTL ограничивает устаревание данных.

## GitHub Actions

GitHub Actions автоматизирует CI: checkout, установка зависимостей, сборка Docker-образов, запуск compose-стенда и проверка контейнеров. Это снижает риск ручных ошибок перед демонстрацией и поставкой.

## EF Core

EF Core является ORM для .NET и подходит для работы IntegrationService или доменных .NET-сервисов с PostgreSQL. Он дает миграции, LINQ, change tracking и управление транзакциями. В микросервисной архитектуре каждый сервис должен владеть своей схемой данных.

## Docker Compose

Docker Compose описывает локальный demo/staging-стенд как код. Он запускает все сервисы, PostgreSQL и Redis в одной сети, задает порты, переменные окружения и зависимости запуска.

## REST vs RabbitMQ

REST хорош для синхронных запросов, когда клиенту нужен ответ сразу: UI открывает карточку ОС, Swagger демонстрирует API, оператор запускает списание. RabbitMQ лучше для асинхронных процессов: длительные операции, интеграция с внешними системами, outbox, retries без удержания HTTP-соединения. В данном проекте REST выбран для понятного demo-сценария, но архитектура может быть расширена событиями.
